Project Management Portal — Architecture, API Structure & Security

Portal: Vue (hosted on Netlify)
Middleware: Netlify Function
Server: NetSuite RESTlet (SuiteApp)

Contents

1. System Architecture Overview

1.1 High-Level Architecture

Vue Portal (Client)
    |
    | HTTPS
    v
Netlify Functions (Middleware API)
    |
    | HTTPS + signed payload (HMAC-SHA256 + timestamp)
    v
NetSuite RESTlet (Server)
    |
    v
Custom Records/Transactions:
- Project (created/maintained in NetSuite)
- Job (created/maintained in NetSuite and Portal)
- Internal Requisition (CRUD via Portal)
Key security principle: Client-side never calls the NetSuite RESTlet directly. It calls the middleware with JWT in the header. The middleware verify the integrity of the JWT, then signs the payload with a unique secret key. This is then sent to the RESTlet which in turn verifies the signature with its own copy of the secret key before authorizing the request. This boundary protects credentials, enforces validation/rate limiting, and blocks unsigned or stale requests.

2. Functional Scope

2.1 NetSuite UI (ERP-managed)

  • Create Projects (custom record)
  • Create Jobs under Projects (custom record)
  • Workflow approvals, CAPEX/OPEX classification

2.2 Portal (Client-side)

  • View Projects (filtered by site/department/role)
  • View Jobs under a Project
  • Create Job under an allowed Project (via middleware + RESTlet)
  • CRUD Internal Requisitions (IRs)
  • Additional portal features: dashboards, summaries, validations (status, thresholds)
Scope guardrail: Portal should not create Projects, nor perform capitalization or journal entries. Keeping write access limited reduces blast radius.

3. Security Model

3.1 Security Flow Overview (Client ↔ Middleware ↔ RESTlet)

  1. Client → Middleware (login): client sends username/password to POST /login. Middleware calls RESTlet users/login with signed payload.
  2. Middleware → Client (login response): on successful login, middleware issues JWT (uid, sid) and returns it to client.
  3. Client → Middleware (protected APIs): client sends Authorization: Bearer <jwt>. Middleware verifies JWT and extracts uid/sid.
  4. SID cache at middleware: middleware caches SID per user for SID_CACHE_TTL_MS (default 5 minutes). If cache is missing/expired/mismatched, middleware sets requireSid=true to force server-side SID validation.
  5. Middleware → RESTlet: middleware injects trusted identity fields (uid, optional sid, requireSid), adds timestamp, computes HMAC-SHA256 signature, and forwards request.
  6. RESTlet validation: RESTlet verifies HMAC + timestamp freshness, resolves principal server-side from uid, and validates SID when required (high-risk actions or requireSid=true).
  7. RESTlet → Middleware → Client: RESTlet returns result, middleware returns sanitized response to client. On successful logout, middleware clears SID cache for that user.

3.2 Core Security Goals

  • Protect secrets: JWT secret, NetSuite shared HMAC secret, and OAuth key material stay server-side only.
  • Authenticate every request: middleware enforces JWT/API-key checks, then RESTlet enforces signature + timestamp checks.
  • Enforce session validity: SID is checked in RESTlet for high-risk actions and when middleware sets requireSid=true.
  • Constrain access: RESTlet derives user access from NetSuite user record and applies site/department authorization.

3.3 Trust Boundaries

Untrusted: browser request body/headers and any client-provided ids/sid.
Trusted only after checks: middleware JWT verification + RESTlet HMAC/timestamp/SID validation.

3.4 Portal → Middleware

  • POST /login does not require JWT and returns JWT + SID on successful NetSuite login.
  • Protected middleware routes require Authorization: Bearer <jwt>.
  • Middleware injects uid from JWT and decides requireSid using SID cache status.

3.5 Middleware → RESTlet Signing

  • HMAC-SHA256 signature over payload without signature (with fresh timestamp).
  • Shared secret is handled as Base64 on both sides.
  • RESTlet rejects stale requests (5-minute window), missing signature, or bad signature.

3.6 SID Validation Strategy

  • Middleware caches SID in Netlify Blobs by uid for SID_CACHE_TTL_MS.
  • High-risk actions always require SID validation in RESTlet (jobs create, ir create/update/delete, users logout/create-access).
  • If cache expired/missing/mismatch, middleware sets requireSid=true and RESTlet validates SID against user management record.
  • On successful logout, middleware clears SID cache entry for that user.

3.7 Purpose of getPrincipal(context, { requireSid })

This function is the RESTlet gatekeeper. It verifies request signature/timestamp, resolves uid to the NetSuite user-management record, optionally validates SID, checks account state (access/lock), and returns normalized identity data used for all downstream authorization decisions.

3.8 API Key Usage (Server-to-Server Only)

  • Middleware also supports x-api-key authentication for integration/non-user flows.
  • API key must never be exposed in frontend code or browser requests.
  • Use API key only when middleware is called from a trusted backend/service.
  • Store API key in server-side secrets manager or environment variables only.

4. Threat Model & Vulnerabilities

6.1 Secret Leakage

Risk: NetSuite tokens/keys leaked via client code, logs, repo, or error traces.
Mitigation: Secrets only in environment variables; never log auth headers; least privilege; rotate tokens; separate dev/stage/prod credentials.

6.2 API Abuse / DoS / Governance Exhaustion

Risk: Flooding middleware endpoints triggers NetSuite calls → governance exhaustion / account slowdown.
Mitigation: rate limiting (IP + user), request size limits, pagination limits, caching of reads, timeouts, circuit breaker on high error rates.

6.3 IDOR (Insecure Direct Object Reference)

Risk: user changes projectId / jobId / irId to access unauthorized data.
Mitigation: server-side authorization checks for every request: site/department/ownership-based access. Do not trust client-provided IDs.

6.4 Injection (SuiteQL / dynamic filters)

Risk: dynamic query concatenation leading to data exposure or unexpected query behavior.
Mitigation: strict allowlist validation; no string-concatenated SuiteQL; sanitize/validate all filters.

6.5 Unauthorized Writes / Business Rule Bypass

Risk: creating IRs for closed jobs/projects; editing processed IRs; bypassing approvals.
Mitigation: RESTlet validates Project status + Job status + IR status before any write.

6.6 Over-Privileged Integration Role

Risk: if the middleware is abused, the attacker can do everything the role can do.
Mitigation: dedicated integration role with only the permissions needed: Projects/Jobs read, IR CRUD, and no GL/JE/FA permissions.

6.7 Replay / Duplicate Submissions

Risk: replayed or delayed requests create duplicate IRs or duplicate lines.
Mitigation: enforce signed timestamp validation window (5 minutes), verify HMAC signature. Add idempotency keys for write operations as a hardening enhancement.

6.8 XSS in Vue

Risk: unescaped input renders scripts, steals tokens, performs actions.
Mitigation: avoid v-html for untrusted data; escape content; use CSP and security headers.

6.9 CSRF (if cookies used)

Risk: malicious site causes user browser to submit requests if cookie-based auth is used.
Mitigation: prefer bearer tokens; otherwise use SameSite=strict + CSRF tokens.

6.10 Input Validation Gaps

Risk: malformed or oversized payloads can trigger unexpected behavior, parser failures, or bypass weak business checks.
Mitigation: strict schema/type/length validation at middleware and RESTlet, plus request body size limits.

6.11 Misconfigured CORS / Browser-Origin Abuse

Risk: broad CORS policies allow untrusted origins to call middleware from browser contexts.
Mitigation: allowlist CORS origins and reject forbidden origins before forwarding to NetSuite.

6.12 Uncontrolled Failure Cascades

Risk: repeated downstream failures (OAuth/RESTlet) cause cascading retries and service degradation.
Mitigation: enforce request timeouts, add circuit-breaker behavior, and monitor error-rate spikes.

6.13 Information Disclosure Through Errors

Risk: stack traces or raw system errors expose implementation details useful to attackers.
Mitigation: return sanitized error payloads and keep detailed traces only in secured logs.

6.14 Overly Broad Integration Permissions

Risk: compromised middleware credentials inherit excessive NetSuite capabilities.
Mitigation: use a dedicated least-privilege integration role scoped to required records/actions only.

5. Logging & Monitoring

6.1 Current Logging Architecture

  • RESTlet has centralized request logging via restlet_handler.
  • Automatic log events are written for every request: API_REQUEST_SUCCESS and API_REQUEST_FAILED.
  • Logs are persisted to custom record customrecord_pm_logs through services/log_service.js.
  • Middleware sends clientInfo with each request (IP + user-agent/header metadata), and RESTlet logs it in custom record field custrecord_pm_log_client_info.
  • Middleware has structured runtime logs (warn/info/error) for auth, rate limit, and upstream failures.

6.2 What is Stored in customrecord_pm_logs

  • Core context: event time, level, event type, source, resource, action, uid, requestId, status.
  • Error context: error code, message, stack (when available).
  • Payload context: sanitized request/response payload snapshots.
  • Client context: clientInfo JSON (IP, user-agent, forwarded-for, origin/referer, language, and client-hint headers when available).
  • Sensitive fields are redacted by log service (e.g. password, signature, token, sid, api key).

6.3 Granular Business Logs

In addition to automatic request logs, RESTlet action branches can add domain-specific breadcrumbs using actionLog (for example business rule violations or state transition blocks). 

actionLog.warn('JOB_STATUS_INVALID_TRANSITION', {
  status: 'FAIL',
  errorCode: 'INVALID_STATUS_TRANSITION',
  errorMessage: 'Cannot move Closed job to In Progress'
});

6. Operational Kill Switches

7.1 Full Shutdown

  • Set RESTlet deployment Deployed = F.
  • Effect: all API operations stop (reads + writes + auth paths through this RESTlet).

7.2 Controlled Write Kill Switch

  • Set script/deployment parameter custscript_pm_enable_writes = F.
  • Effect: write actions are blocked server-side; read operations can continue.

7.3 User Forced Logout

  • Clear user session hash in customrecord_pm_usermanagement.custrecord_pm_usermngt_session.
  • Effect: user is invalidated, but enforcement may not be immediate due to SID cache TTL in middleware.
  • Practical note: invalidation is guaranteed on subsequent server SID validation (high-risk actions or cache expiry).

7. API Calls Reference

Middleware calls the RESTlet using POST JSON with resource + action. Reads and writes are both handled in the RESTlet POST route for middleware integration. 

Authentication and identity fields sent by middleware:
{
  "uid": "1001",
  "sid": "sessionHashIfAvailable",
  "requireSid": true|false,
  "timestamp": 1739870000000,
  "signature": "hex_hmac_sha256"
}
Identity resolution model: middleware sends uid (plus auth/signature fields). RESTlet then resolves server-side user context via getPrincipal() (email, subsidiary, sites, departments, roles, permission codes). If frontend/client sends principal-like identity fields, they are not trusted; authorization relies on server-resolved data only.

8. Developer Guide: Adding New Actions

This is a concrete playbook for adding a new action. Follow each step in order.

8.1 Where to edit (file map)

  • src/FileCabinet/SuiteApps/com.netsuite.pm/BC_PM_RST_API.js: route/action wiring, SID requirement, authorization checks.
  • src/FileCabinet/SuiteApps/com.netsuite.pm/lib/validators.js: payload validation for the new action.
  • src/FileCabinet/SuiteApps/com.netsuite.pm/services/job_service.js (or relevant service): business logic + record write/read.
  • src/FileCabinet/SuiteApps/com.netsuite.pm/lib/authz.js: permission policy mapping in canPerform if needed.
  • technical-doc.html: update Section 7 API reference after implementation.

8.2 Example Scenario: Add a New Action to Job Resource (Update Status)

For example, if we need to add a new action for resource jobs to update status (action name: update-status), follow the steps below.

  1. Add validator in lib/validators.js. 
    const validateJobUpdateStatus = (body) => {
  if (!body || typeof body !== 'object')
    throw errors.badRequest('INVALID_BODY', 'Body must be JSON object');
  return {
    id: asInt(body.id, 'id'),
    status: asInt(body.status, 'status')
  };
};
  2. Export validator in same file return block. 
    return {
  ...
  validateJobUpdateStatus
};
  3. Add service method in services/job_service.js. 
    const updateJobStatus = ({ id, status }) => {
  record.submitFields({
    type: C.REC_JOB,
    id,
    values: { [C.FLD_JOB_STATUS]: status }
  });
  return getJob(id);
};
  4. Export service method from job_service.js. 
    return { listJobsByProject, getJob, createJob, updateJobStatus };
  5. Update restlet in BC_PM_RST_API.js, inside resource === 'jobs'.

    5.1 Update route/action:

    const action = V.oneOf(String(context.action || ''), 'action', ['create', 'update-status']);
const principal = authz.getPrincipal(context, {
  requireSid: shouldRequireSid({ method: 'POST', resource: 'jobs', action, context })
});

if (action === 'update-status') {
  if (!authz.canPerform(principal, 'JOB.UPDATE_STATUS'))
    throw errors.forbidden('NO_ACCESS', 'Not allowed');

  const payload = V.validateJobUpdateStatus(context);
  return ok(jobs.updateJobStatus(payload));
}

    5.2 Set high-risk behavior (HIGH_RISK_ACTIONS policy):

    In BC_PM_RST_API.js, update HIGH_RISK_ACTIONS. For write actions, add them as high-risk so SID is always validated server-side. 

    const HIGH_RISK_ACTIONS = {
  jobs: ['create', 'update-status'],
  irs: ['create', 'update', 'delete'],
  users: ['logout', 'create-access']
};
  6. Logging

    Automatic request logs already exist for every RESTlet request: API_REQUEST_SUCCESS and API_REQUEST_FAILED.

    (Optional) If you need granular logs for business-level events, you can use actionLog in RESTlet action branches: 

    actionLog.warn('JOB_STATUS_INVALID_TRANSITION', {
  status: 'FAIL',
  errorCode: 'INVALID_STATUS_TRANSITION',
  errorMessage: 'Cannot move Closed job to In Progress'
});

    If you need the logger in service files, pass actionLog from the restlet as a parameter to the action function. 

    // BC_PM_RST_API.js
if (action === 'update-status') {
  const payload = V.validateJobUpdateStatus(context);
  return ok(jobs.updateJobStatus(payload, actionLog));
}

// services/job_service.js
const updateJobStatus = ({ id, status }, actionLog) => {
  try {
    record.submitFields({
      type: C.REC_JOB,
      id,
      values: { [C.FLD_JOB_STATUS]: status }
    });
    return getJob(id);
  } catch (e) {
    if (actionLog) {
      actionLog.error('JOB_UPDATE_STATUS_FAILED', {
        status: 'FAIL',
        errorCode: e.code || 'JOB_UPDATE_STATUS_FAILED',
        errorMessage: e.message || 'Failed to update job status'
      });
    }
    throw e;
  }
};

    - Event-specific fields you should pass: status, errorCode, errorMessage

    - The following fields are optional because they are already included as part of the automated logging: requestPayload, responsePayload, stack

    - These base fields are included and prefilled when actionLog is used within Restlet requests or pass as parameter from Restlet to Service: resource, action, uid, requestId, source, clientInfo

  7. Add permission mapping (optional but recommended) in lib/authz.js policy map. 
    'JOB.UPDATE_STATUS': ['JOB.UPDATE_STATUS']

    Permission mapping in authz.js does not create permissions. You must also:

    1. Add the permission code to customlist_pm_perms.
    2. Assign it to a role in customrecord_pm_roles (custrecord_role_perms).
    3. Permissions are resolved into principal.permissionCodes by authz.getPrincipal().
  8. Update documentation: add endpoint/payload/response under Section 7.2 Jobs.

8.3 Minimum checklist for every new action

  • Validation exists and is strict (no raw context passthrough).
  • Authorization check exists (canPerform and/or project access checks).
  • SID policy considered (high-risk action added to HIGH_RISK_ACTIONS).
  • Service returns normalized payload, not raw record objects.
  • Section 7 API documentation updated.

9. Developer Guide: Adding New Resources

Use this when you need a new domain/resource (not just a new action under an existing resource).

9.1 When to create a new resource

  • Create a new resource when it has its own domain boundary, validation, authorization rules, and service logic.
  • Do not use a generic catch-all resource for unrelated domains; that weakens routing clarity and security checks.

9.2 Step-by-step (resource scaffold)

  1. Create service file in services/<resource>_service.js with explicit methods per action.
  2. Add validators in lib/validators.js for each action payload.
  3. Wire RESTlet routes in BC_PM_RST_API.js: add a new if (resource === '<resource>') branch and explicit action handling.
  4. Set SID policy: add write/high-risk actions in HIGH_RISK_ACTIONS so server-side SID validation is always enforced.
  5. Add authorization mapping in lib/authz.js policy map (canPerform).
  6. Add logging: rely on automatic request logs and optionally add business-level events via actionLog.
  7. Update API docs: add the new resource and actions in Section 7 API reference.

9.3 Minimum checklist for every new resource

  • Resource has dedicated service methods (no mixed domain logic).
  • Each action has validator coverage.
  • Authorization and SID policy are explicitly defined.
  • Error codes/messages are consistent with existing response format.
  • Section 7 API documentation is updated.
