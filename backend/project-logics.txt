Statement of Work (SOW) is a document that contains detail about a project to be undertaken.
Sales order is Netsuite default transaction.
Project is a custom record.
Project Phase is a child record of parent further breaking the project into phases.
Revenue Plan is a custom record which is link to a project phase.

---------------------------------------------

SOW is drafted and signed.
Sales order is created based on the SOW.
Amendments can happen in SOW which can lead to amendments in the Sales Order and thus project phases quantity can be updated or new project phases are created under the same project.
SOW can be extended which can lead to creation of a different sales order (still linked to the same project).
One project can be associated with one or many sales orders (SO). One SO belongs to one project only.
The Sales Order field used to link to a project is: cseg_project_seg.
Because this segment value is not the project internal id, backend must resolve links by comparing segment text to project name.

Milestones:
- The lines on a Sales Order represent the milestones of the project.
- Each milestone is represented by a service item (line item), BUT the important piece for users is the Sales Order line description.
- We should not rely on "one service item per milestone" (unlikely). Instead, we treat a milestone as unique by:
  (service item id + line description) together.

Phases:
- A milestone can have many phases.
- Project phases NO LONGER reference a Sales Order directly.
  Whenever we need to know Sales Orders for a project, backend must look up all SOs where LOWER(cseg_project_seg text) equals LOWER(project name).
- Phases still have a dropdown field in NetSuite for milestone (service item):
  custrecord_pm_phase_milestone
  However, what matters for the UI is the line description. A new text field is introduced:
  custrecord_pm_phase_milestone_desc
- In the frontend Create/Edit Phase form, we show one dropdown called "Milestone" and display only the line description.
  Backend must build the dropdown options by scanning all Sales Order lines for the project and mapping:
  (service item id + line description) -> display label (line description).
  When the form is submitted, frontend must send BOTH:
  - milestone (service item) id
  - milestone description (line description)
  so backend can set:
  custrecord_pm_phase_milestone = <service item id>
  custrecord_pm_phase_milestone_desc = <line description>

Contract value / variance / flags:
- All the project phases total amount (grouped by department) must exactly match the total amount of all SO lines (grouped by department) across ALL SOs linked to the project.
- Backend computes:
  - variance amount
  - a department-wise summary (SO totals vs Phase totals)
  - mismatch flag(s)
  Frontend should stay dumb and only render the values.

Project status rules (only these matter):
- Draft (default when projects are created)
- Active
- Completed
- Close

Active status can only be set once there is NO variance between SO totals and Phase totals by department.
If a project is already Active but later a variance is introduced (SO amended, phases changed, etc):
- keep the status as Active
- BUT disallow creation of revenue plans
- AND disallow modification of existing plans until discrepancies are resolved

Because these are backend-driven rules, after every action (create/edit phase, create/edit plan, etc.)
we must refetch the backend-computed flags so the UI remains consistent.

---------------------------------------------

Revenue plan logics are as follow:

If an invoice is created for the sales order, the double entry will be:
Dr T.Receivables
Cr Deferred Income

And when we submit the revenue plan for the current month, it needs to create a journal:
Dr Deferred Income
Cr Revenue

If no invoice was yet created, the double entry would be:
Dr Unbilled Receivables
Cr Revenue

Lets have a proper scenario:

If we have a project with the following phases for a duration of 12 months starting Jan 2025 to Dec 2025:
Phase 1 - Dept A - Qty 10 @ rate $500
Phase 2 - Dept A - Qty 4 @ rate $1000
Phase 3 - Dept B - Qty 5 @ rate $100

We will have the revenue plan as follows:
For each month, we would create a column. The current month will be actual and those months after it are forecast.

Each month (on the actual column), user manually entered how many qty has been completed.

Lets say Jan and Feb there was nothing, now we are in March 2025. Here is the qty breakdown per phase for the actual month:
Phase 1 - Qty 2 - $1000
Phase 2 - Qty 0 - $0
Phase 3 - Qty 1 - $100

Also lets assume there was an invoice created last month for Phase 1, Dept A of qty 2 and total $1000.

When in the actual month we are done inserting our quantities and we do submit. Finance user will come and click a button to generate the rev rec journal:

Debit Deferred Income $1000
Credit Revenue $1000
Debit Unbilled Receivable $100
Credit Revenue $100

Now we are in April 2025. The qty breakdown is:
Phase 1 - Qty 3 - $1500
Phase 2 - Qty 1 - $1000
Phase 3 - Qty 0 - $0

There has been one invoice in the current month for Phase 3, Dept B of qty 2 and total $200.

When the rev rec journal is created, we have the follwing entries:

The previous journal is reversed completely:

Debit Revenue $1000
Credit Deferred Income $1000
Debit Revenue $100
Credit Unbilled Receivable $100

Then the current month rev rec journal is created:

Debit Deferred Income $1000
Debit Unbilled Receivable $1500
Credit Revenue $3500
Debit Deferred Income $100
Credit Revenue $100

The reversal journal should be handled automatically using Netsuite journal reversal option.


---------------------------------------------

In frontend, Revenue plan will be in a tabular format where the phases will be rows and there will be all the months as columns clearly indicating the Plan Type which are actual months (those that have past), the current month and the forecast months.

Behind the scenes, in Netsuite, this is a custom record. The record instances will be the phase and the period (custrecord_pm_revplan_period, last day of month).
To know how many months column to have, we need to calculate the number of months between the Project Start Date and Project End Date.
So for example, if we have a project with Phase 1 and Phase 2 with start date 01/10/2025 and end date 31/12/2025 (duration is 3 months). We will have revenue plan (record instances) as follows (assuming current month is Oct 2025):

1 - Phase 1 (Oct 2025) Plan Type = Actual
2 - Phase 2 (Oct 2025) Plan Type = Forecast
3 - Phase 1 (Nov 2025) Plan Type = Forecast
4 - Phase 2 (Nov 2025) Plan Type = Forecast
5 - Phase 1 (Dec 2025) Plan Type = Forecast
6 - Phase 2 (Dec 2025) Plan Type = Forecast

We have only two Plan Types: Actual and Forecast.
We have Revenue plan status which is Open, Rev Rec Ready, Completed, Cancelled.

Open - User can still input quantity, finance user cannot generate rev rec journal
Rev Rec Ready - User cannot input quantity, finance can now generate rev rec journal
Completed - Rev rec journal is created. User may update quantity but no rev rec journal will be re created for that month.
Cancelled - Yet to be defined.

In the frontend:

User will be able to input quantity in the months cells (be it of any plan type: actual, current, forecast).

** Current is not really a Plan Type, its just an Actual Plan Type but frontend will display it as current so user knows this is the current month we are on.

Revenue Plan custom record date / period changes:
- We no longer have separate month/year fields on Revenue Plan.
- We now store a single Date field: custrecord_pm_revplan_period
- This date should always be the LAST DAY of the month.
  Example: "Apr 2026" becomes 30/04/2026.
- When we need month/year, we extract them from custrecord_pm_revplan_period.

Current period determination change:
- We no longer determine the current month from the accounting period.
- We determine "current month" using new Date() and comparing it to custrecord_pm_revplan_period.
- Timezone handling: we use the COMPANY preference timezone (not each user's preference timezone)
  so that "current month" is consistent for everyone (especially around month boundaries).

Quantity entered can be decimals and can be negative. They will have a hard limit based on the project phase defined quantity. So if Phase 1 has a quantity of 10 (lets call it defined qty).

When we entered a qty in a cell, we need to ensure if we sum all the quantities entered for all columns for that row, the sum should be >= 0 and <= Defined Qty.

For project phase, we need a validation:

Because defined quantity can change at any time in the phase duration.
We need to have the following check:
 - If defined quantity is being changed to less than what it was previously, we need to check that the sum of qty of all revenue plans with plan type actual is not greater than the new defined qty we want to set for that phase.
 Lets say Defined qty was 3 previously, now we want to change it to 1.
 And the sum of all actual qty is 2. We should not allow the phase quantity to be changed and giving a warning to correct the revenue plan actual quantities so that its not greater than the new defined qty we want to set.


---------------------------------------------

Project Phase will have the following statuses:

Pending - No rev plan exist for the phase.
Require Rev Plan Update - The number of months in the rev plan is not equal to how many months there are between the project start date and end date.
Rev Plan created - The rev plan exist for the phase and has the right amount of months per the project start date and end date.
